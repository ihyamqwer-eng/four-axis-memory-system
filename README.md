# Three-Axis Memory System

一个将「记忆」拆解为**中心轴、赫尔墨斯轴、对象轴**三个维度的个人 AI 知识管理系统。

不是向量数据库的替代品——而是它的上层组织框架。

---

## 核心思想

**三轴 ≠ 三种数据库。** 三轴是把同一套记忆数据，按三种不同的查询模式分别组织和索引——**事件驱动 vs 状态驱动**。

将记忆按**用途**而非格式分离：

| 轴 | 驱动属性 | 类比人脑 | 写入触发 | 可覆盖 |
|----|----------|----------|----------|--------|
| **中心轴** (Center) | **事件驱动** — 记录"发生了什么"和"原话怎么说" | 回忆往事 + 对话录音 | 外部事件发生 / 每轮对话 | ❌ 只追加 |
| **赫尔墨斯轴** (Hermes) | **状态驱动** — 记录"我是谁"和"规则是什么" | 现在的状态 + 行事准则 | 状态变迁 / 规则更新 | ✅ 最新覆盖旧 |
| **对象轴** (Object) | **状态驱动** — 记录"事情进度" | 项目进展到哪了 | 任务推进 | ❌ 只追加 |

**关键区分：**
- **事件驱动**（中心轴）：发生了什么就记什么，不合并、不覆盖、不修改。每一条都是独立事件。原来两套时间线（总轴 + 会话轴）合并为一条统一的、按时间排序的事件流——写论文、发消息、状态变更，按同一个时钟排列。
- **状态驱动**（赫尔墨斯轴、对象轴）：关注的是"当前是什么状态"。赫尔墨斯轴用 last-write-wins（新的覆盖旧的）+ 规则引擎 + skill 映射 + 系统快照；对象轴用追加 changelog。

**为什么这种区分重要？** 单纯的向量检索分不清"三年前我买了件衣服"（事件）和"我刚才吃了饭"（另一个事件）的区别——它们都只是"买衣服"和"吃饭"的向量。三轴通过不同的持久化策略，让"回忆过去"和"查询现状"走不同的路径，互不干扰。

**为什么从四轴减到三轴？** 因为事件驱动的时间线和会话在实践中共用同一套读写模式——只追加、按时间检索、永不修改。与其让 Agent 在"总轴"和"会话轴"之间做无意义的二选一，不如合并为一条统一事件流，减少维度、降低心智负担。赫尔墨斯轴则在上帝轴的基础上多了 rules engine、skill mapping 和 system snapshots——不只是"我是谁"，还包括"我怎么做"。

## 三层架构

```
第一层：三轴（原始数据层 — 所有记忆的源）
  中心轴 · 赫尔墨斯轴 · 对象轴
  （每种轴用不同的持久化策略）
         ↓
第二层：语义检索（LanceDB + Embedding）
  extract-fact → 向量化 → 语义搜索
  （从三轴中提取重要事实，做跨维度检索）
         ↓
第三层：知识库（Wiki Vault — 可读知识沉淀）
  concepts · syntheses · entities · sources
  （重要度≥4 的事实自动沉淀为 Wiki 条目）
```

## 查询路径：大多数查询不需要语义搜索

三轴最直接的效率来源：**按轴查询 = 直接读文件，零计算。**

```
"最近发生了什么 / 他当时怎么说的"
                                → 中心轴.jsonl 最后N条         ← 纯文件读取
"我现在是什么状态 / 有哪些规则"
                                → 赫尔墨斯轴/xxx.jsonl 最后1条  ← 纯文件读取
"项目进展到哪了"                → 对象轴/xxx.jsonl 最后1条      ← 纯文件读取

"我记得好像有个什么事..."       → LanceDB 语义搜索兜底         ← 只有这个需要向量
```

语义搜索（第二层）只在**跨轴模糊查询**时启用——你不知道该查哪个轴、只记得大概内容。日常"查事件"、"查状态"、"查项目进度"都在轴层面直接完成，不需要 Embedding，不需要 LLM。

**"三轴是向量库的上层组织框架"** 的具体含义就是这个：90% 的记忆查询在轴层面解决，只有剩下的 10% 需要向量兜底。

核心数据流：
```
用户消息
   ↓ 写入中心轴（统一事件流 — 原始消息 + 外部事件，按时间序追加）
   ↓ 自动评估
       状态变化 → 写入赫尔墨斯轴（last-write-wins + rules engine + system snapshot）
       重要事件 → 写入中心轴（同一条事件流，不重复）
       任务推进 → 写入对象轴（changelog）
   ↓ extract-fact（LLM 分析 → 向量化 → LanceDB）
   ↓ importance≥4 → 自动沉淀为 Wiki 条目
```

## 设计原则

1. **只追加，不修改** — 中心轴和对象轴的历史可追溯。过去写下的任何内容都不会被修改。
2. **按职责分离，按需检索** — 查事件走中心轴，查状态+规则走赫尔墨斯轴，查进度走对象轴。不让一个维度扛所有查询。
3. **Agent 原生，不依赖外部服务** — 三轴运行在 AI Agent 内部，JSONL 文件是直接持久化层，不需要独立数据库。
4. **自动沉淀，由点及面** — 原始事实存三轴 → 重要事实自动向量化 → 累积到阈值自动成文。知识从对话中自然生长。
5. **规则与状态同源** — 赫尔墨斯轴不只存状态，还存 rules engine 的规则映射和 skill 配置。Agent 想查"我该怎么做"和"我现在是谁"走同一个地方。
6. **少一个维度，少一次思考** — 三轴取代四轴，减少了"这条该写总轴还是会话轴"的决策成本。所有事件驱动的内容统一归中心轴，Agent 每次只需要判断：这是事件、状态、还是进度。

## 应用场景

### 适配场景

| 场景 | 适合程度 | 说明 |
|------|----------|------|
| **个人 AI 助手/陪伴** | ⭐⭐⭐⭐⭐ | 需要记住你的习惯、偏好、说过的话；聊天中穿插"上次你说..."自然又贴心 |
| **AI 伴侣/朋友/聊天机器人** | ⭐⭐⭐⭐⭐ | 长程对话的记忆根基：记住共同经历、别问同样的问题、状态驱动能让陪伴感更强 |
| **长期对话的上下文管理** | ⭐⭐⭐⭐⭐ | 不限 token 窗口、不依赖一次性 prompt、真正的"它会记得" |
| **个人知识笔记系统** | ⭐⭐⭐⭐ | 三轴提供结构化组织，Wiki 层沉淀可读知识，适合做第二大脑 |
| **AI 角色扮演/人设保持** | ⭐⭐⭐⭐ | 赫尔墨斯轴保持角色状态和规则一致性，对象轴跟踪故事线 |
| **个人项目/任务管理** | ⭐⭐⭐⭐ | 对象轴天然适配轻量项目管理 |
| **共享记忆（跨渠道）** | ⭐⭐⭐ | 支持多渠道合并，但单用户设计 |
| **企业级/团队使用** | ⭐⭐ | 单用户架构，无分布式支持 |

### 不适合场景
- 高并发读写
- 多租户 SaaS
- 严格 ACID 事务需求
- 已有成熟向量库 + 图数据库 + 传统数据库的基础设施团队（他们不需要另一层框架）

### 四轴 vs 三轴

| | 四轴（旧） | 三轴（本文） |
|---|---|---|
| **事件轴数量** | 两个（总轴 + 会话轴），Agent 需判断"这条写哪" | 一个（中心轴），统一事件流 |
| **状态轴** | 上帝轴 — 只有状态 | 赫尔墨斯轴 — 状态 + rules engine + skill mapping + system snapshots |
| **决策成本** | 高 — 每次更新要选四个轴 | 低 — 每次更新只判断：事件、状态、还是进度 |
| **对象轴** | 同 | 同 |
| **中心轴开销** | 无 | 略高（单轴负责所有事件，但读写量不大，开销可忽略） |

**三轴的核心差异化优势：**
1. **事件驱动 vs 状态驱动的双策略分离** — 其他方案要么全追加（膨胀快）、要么全覆盖（丢历史），三轴按"这个数据是历史事件还是当前状态"分别采用不同策略。
2. **三层沉淀链路** — 从原文到向量到 Wiki，自动提炼，无需手动管理。
3. **Agent 原生** — 三轴直接运行在 AI Agent 内部，Agent 在回复循环中直接读写 JSONL。不需要独立数据库、容器编排或微服务。
4. **专为"回忆"和"陪伴"设计** — 不是通用知识管理，而是"A 和 B 之间共同经历的事情"。时间线感、状态一致性、对话回溯是核心诉求。
5. **查询不走向量** — 90% 的记忆查询在轴层面直接读文件完成，语义搜索只在跨轴模糊查询时兜底。
6. **一条命令写出所有轴** — `auto-update` 命令自动评估每个轴是否需要更新，无需手动判断。
7. **赫尔墨斯轴一体化** — 状态、规则、技能映射、系统快照放在同一个轴里，Agent 查一次就知道"我现在是什么、我该怎么做、我会什么、我经历了什么变化"。

## 局限

- 单用户场景设计，无多租户支持
- 语义检索依赖外部 LLM / Embedding 模型
- 字段未形式化定义（工程语言而非学术语言）
- 未做大规模负载验证
- 中心轴将时间线和对话合并后，单轴事件量会增长更快——但 JSONL append 的写入成本极低，实践中不构成瓶颈
